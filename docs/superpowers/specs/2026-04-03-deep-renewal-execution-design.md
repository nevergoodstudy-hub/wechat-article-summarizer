# 深度焕新执行设计（全量并行 + 稳健型）

日期：2026-04-03  
来源：`docs/plans/DEEP_RENEWAL_FEASIBILITY_PLAN.md`

## 1. 目标与约束

### 目标
在不牺牲可回滚性与可验证性的前提下，推进四轨并行焕新：
- 架构轨（边界收口、防回胖、并发结构化）
- 安全轨（MCP 入参校验、SSRF 防护、审计脱敏）
- 工程轨（ruff/mypy/pytest/security smoke 统一门禁）
- 仓库治理轨（噪音目录隔离、核心视图建立）

### 约束
- 单批次改动可在 1~3 天完成
- 每批次必须具备：代码变更 + 自动化验证 + 可回滚点
- 不通过统一门禁的变更禁止合并

## 2. 执行模型

采用“轨道并行 + 统一门禁 + 小批迭代”模型：
1. 四轨并行推进，但每批次只改一个清晰目标
2. 统一门禁优先于功能扩展
3. 所有安全变更必须配套测试
4. 所有结构变更必须验证边界不回退

## 3. 首批落地顺序（已确认）

### Batch 1：仓库治理
- 目标：建立噪音隔离规则和核心代码视图
- 输出：目录/工具配置调整，减少无关扫描干扰

### Batch 2：工程门禁
- 目标：统一本地与 CI 的质量入口
- 输出：ruff/mypy/pytest/security smoke 的一致执行链路

### Batch 3：MCP 安全基线
- 目标：统一入参校验中间层
- 输出：高风险工具参数默认拒绝 + 自动化测试

### Batch 4：SSRF 防护
- 目标：补齐 DNS Rebinding 与重定向安全
- 输出：DNS 固定解析策略、跳转校验、异常 IP 表示拦截、测试

## 4. 第二阶段入口条件

仅当 Batch 1~4 满足以下条件时，进入结构焕新：
- CI 门禁稳定可用
- MCP 与 SSRF 核心安全测试通过
- 核心目录扫描效率显著提升

第二阶段任务：
- GUI 防回胖治理
- 领域边界守卫脚本 + CI 阻断
- 并发模型迁移（TaskGroup + 限流 + ExceptionGroup）
- 结构化日志与审计脱敏落地

## 5. 验收口径

每批次验收统一口径：
- 功能正确：需求对应点被实现
- 质量通过：ruff/mypy/pytest/security smoke 全绿
- 可回归：新增或更新测试覆盖关键路径
- 可回滚：变更集中、边界清晰

## 6. 风险控制

- 风险：并行改动冲突上升  
  缓解：批次小型化，统一门禁先行

- 风险：安全修复碎片化  
  缓解：统一中间层 + 安全 smoke

- 风险：架构改造后回退  
  缓解：边界守卫脚本 + CI 阻断

## 7. 当前执行决策

已确认策略：**全量并行（D）+ 稳健型（A）+ 轨道并行/统一门禁（推荐方案）**。
