## 0. 执行原则

1. **小批次可回滚**：每批次 1~3 天，单一目标，避免跨主题大改。  
2. **统一质量门禁**：每批次必须通过 `ruff + mypy + pytest + security smoke`。  
3. **先底座后重构**：先做仓库治理与门禁，再做结构与并发演进。  
4. **安全变更必带测试**：MCP/SSRF/审计相关改动必须同时补充测试。

---

## 1. 批次总览（Batch 1~4）

## Batch 1：仓库治理（噪音隔离 + 核心视图）

### 目标
降低全仓噪音，建立“业务核心目录”关注面，提升检索与评审效率。

### 计划任务
- 1.1 盘点并确认噪音目录基线：`.mypy_cache/`、`.tmp_skillhub_sync/`、`.warp/`、`target/`、`apps/desktop/node_modules/` 等。  
- 1.2 校验 `.gitignore` 覆盖完整性，补齐遗漏项。  
- 1.3 新增“核心视图规则”文档（明确本阶段改造默认范围：`src/wechat_summarizer`、`tests`、`.github/workflows`、`pyproject.toml`）。  
- 1.4 约定开发命令仅扫描核心目录，避免全仓扫描。

### 验收标准
- 噪音目录规则清晰并可执行。  
- 团队可直接按核心目录执行检查命令。

---

## Batch 2：工程门禁统一（本地/CI 一致）

### 目标
建立单一质量入口，消除“本地通过 CI 失败”差异。

### 计划任务
- 2.1 提供统一质量命令入口（脚本或 make/task 方案，兼容 Windows/POSIX）。  
- 2.2 CI 对齐统一入口（至少在 `lint/test/security` 三类 job 中复用同一命令语义）。  
- 2.3 将 `security smoke` 纳入默认校验链（最小可行场景，不阻塞全部功能迭代）。  
- 2.4 补充失败输出规范（便于快速定位：lint/type/test/security）。

### 验收标准
- 本地与 CI 触发同一套检查逻辑。  
- 新 PR 必须经过统一门禁。

---

## Batch 3：MCP 安全基线（统一入参校验）

### 目标
在 MCP 工具层实施“默认拒绝 + 明确放行”的参数策略。

### 计划任务
- 3.1 识别 MCP 工具入口与参数注入点。  
- 3.2 建立统一参数校验中间层（长度、字符集、路径白名单、URL 协议限制等）。  
- 3.3 将工具逐步接入中间层，避免散落式校验。  
- 3.4 编写安全测试：非法输入拒绝、合法输入放行、错误信息不泄露敏感细节。

### 验收标准
- MCP 工具统一走校验层。  
- 安全测试覆盖关键高风险参数。

---

## Batch 4：SSRF 防护增强（含 DNS Rebinding）

### 目标
实现 SSRF 防护闭环：解析、连接、重定向、地址表示法、元数据地址拦截。

### 计划任务
- 4.1 梳理所有外部请求路径与 HTTP 客户端封装点。  
- 4.2 引入“解析后固定连接”策略，防 DNS Rebinding。  
- 4.3 禁止自动重定向或实现逐跳校验。  
- 4.4 拦截变体 IP 表示（十进制/八进制/IPv6-mapped 等）与云元数据地址。  
- 4.5 补齐 SSRF 回归测试集（允许/拒绝样例）。

### 验收标准
- SSRF 关键攻击面具备自动化防护与测试。  
- 默认策略为最小权限访问。

## 3. 每批次固定执行清单（DoD Checklist）

- 变更说明清晰（目标、范围、风险）  
- 代码改动小且聚焦  
- 自动化检查通过（ruff/mypy/pytest/security smoke）  
- 关键路径测试已补齐/更新  
- 可回滚（无跨批次强耦合）
