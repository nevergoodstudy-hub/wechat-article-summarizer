## 优先级说明
- **P0 — 紧急**: 阻断性问题，直接影响功能可用性或存在严重安全风险，必须立即修复
- **P1 — 高优先级**: 重要架构/安全缺陷，影响可维护性和安全性，应在下一版本修复
- **P2 — 中等优先级**: 代码质量和规范问题，影响长期可维护性，应在近期迭代中解决
- **P3 — 低优先级**: 改进建议，提升项目成熟度，可按需排期

## P0 — 紧急问题 (4项)

### P0-1: DI容器初始化阻断测试套件
- **来源**: BUG-001, Round 1 & Round 3 研究
- **问题**: `container.py` 在模块导入时急切初始化所有服务（包括LLM API客户端、httpx连接池等外部依赖）。任何导入容器模块的测试文件都会触发外部连接超时，导致186个测试被阻断。
- **影响**: 387个测试用例中仅201个可执行（52%通过率），开发效率严重受损。
- **根因**: 容器使用急切初始化模式而非惰性初始化；缺少测试用覆盖机制。
- **参考**: dependency-injector 库的 `providers.Singleton` 惰性模式，pytest `conftest.py` 容器覆盖模式。

### P0-2: GUI app.py 113KB 上帝对象
- **来源**: Round 1 研究 (God Object反模式)
- **问题**: `src/wechat_summarizer/presentation/gui/app.py` 文件大小为113KB（约3000+行），包含视图构建、事件处理、业务逻辑、状态管理等所有GUI责任。
- **影响**: 无法独立测试GUI组件；新功能添加极其困难；任何修改都可能产生意想不到的副作用；严重阻碍团队协作。
- **根因**: 增量开发中每个新功能都被添加到主类，缺少定期重构。
- **参考**: CustomTkinter MVVM模式，Extract Class重构策略，Mediator模式。

### P0-3: SSRF防护缺少DNS重绑定保护
- **来源**: WARN-005, Round 2 研究
- **问题**: URL验证和实际HTTP请求使用独立的DNS解析。攻击者可设置TTL=0的DNS记录，在验证阶段解析为合法IP，在请求阶段解析为内部IP（如127.0.0.1, 169.254.169.254）。
- **影响**: 攻击者可通过精心构造的URL绕过SSRF保护，访问内部服务/云元数据端点。
- **根因**: 采用了"验证时解析+请求时再解析"的TOCTOU模式，而非"解析一次，验证IP，强制连接该IP"的安全模式。
- **参考**: AutoGPT GHSA-wvjg-9879-3m7w, BentoML CVE-2025-54381, OWASP SSRF防护指南。

### P0-4: MCP服务器缺少输入验证/命令注入风险
- **来源**: Round 2 研究 (MCP安全)
- **问题**: MCP服务器（25KB `server.py`）处理外部工具调用时，如果参数未经充分清理就传递给系统命令或文件操作，存在命令注入风险。
- **影响**: 恶意MCP客户端可通过注入payload执行任意系统命令。
- **根因**: MCP协议设计之初以功能为先，安全机制不完善；工具参数缺乏严格验证。
- **参考**: Palo Alto Unit42 MCP攻击向量研究, CVE-2025-66416, Equixly MCP安全分析。

## P1 — 高优先级问题 (8项)

### P1-1: 领域层边界违反
- **问题**: 领域层（`domain/`）中存在对基础设施层的导入引用，违反六边形架构核心规则：领域层不应依赖任何外部包。
- **影响**: 领域逻辑与具体实现耦合，无法独立测试领域模型，架构退化风险。
- **参考**: 六边形架构端口/适配器规则，DDD领域模型隔离原则。

### P1-2: 缺少结构化并发 (asyncio.TaskGroup)
- **问题**: 项目使用 `asyncio.gather()` 进行并发操作，而非Python 3.11+的 `asyncio.TaskGroup`。gather在任务失败时不会取消其他任务，可能留下孤立任务。
- **影响**: 并发错误处理不可靠；孤立任务可能导致资源泄漏；不符合Python 3.14最佳实践。
- **参考**: Python 3.14 asyncio文档，PEP 654 ExceptionGroup。

### P1-3: MCP审计日志泄露敏感数据
- **来源**: WARN-003
- **问题**: MCP操作的审计日志未充分清理敏感信息（API密钥、用户数据等可能出现在日志中）。
- **影响**: 日志文件泄露可能暴露用户凭据和API密钥。
- **参考**: MCP安全最佳实践 — 结构化日志条目，敏感字段脱敏。

### P1-4: SSRF无HTTP重定向验证
- **问题**: 当外部URL返回HTTP 301/302重定向到内部IP时，SSRF检查不会对重定向目标进行验证。
- **影响**: 攻击者可设置外部URL重定向到 `http://127.0.0.1/internal-api` 绕过防护。
- **参考**: OWASP SSRF Cheat Sheet — 重定向验证要求。

### P1-5: 替代IP表示法未被阻止
- **问题**: SSRF保护仅检查标准格式IP，未覆盖十进制表示(2130706433)、八进制(0177.0.0.1)、IPv6映射IPv4(::ffff:127.0.0.1)等替代格式。
- **影响**: 可通过替代IP格式绕过黑名单。
- **参考**: BentoML CVE-2025-54381补丁绕过案例。

### P1-6: PBKDF2盐值实现需验证
- **来源**: WARN-002
- **问题**: `security.py` 中 `_get_or_create_salt()` 需要确认：盐值是否每个凭据独立生成、长度是否≥16字节、存储是否安全。历史版本使用固定盐值。
- **影响**: 若盐值仍为固定或共享，彩虹表攻击可破解密码/密钥。
- **参考**: NIST SP 800-132 PBKDF2建议。

### P1-7: MCP服务器无沙箱/权限限制
- **问题**: MCP服务器运行时拥有与宿主进程相同的文件系统和网络访问权限，无沙箱隔离。
- **影响**: 恶意或被攻破的MCP客户端可通过工具调用访问任意系统资源。
- **参考**: MCP规范安全最佳实践 — 使用容器/chroot/应用沙箱限制访问。

### P1-8: 52%测试不可独立执行
- **来源**: WARN-004
- **问题**: 超过半数的测试依赖于特定执行顺序或共享可变状态，无法独立运行。
- **影响**: 无法可靠地单独运行失败测试进行调试；CI并行执行可能产生不确定结果。
- **参考**: pytest fixture作用域管理，测试隔离原则。

## P2 — 中等优先级问题 (10项)

### P2-1: 4711个ruff代码质量问题
- **来源**: WARN-001
- **问题**: ruff扫描发现4711个代码风格/质量问题，其中83%可自动修复。
- **影响**: 代码一致性差，可读性降低，潜在bug风险。
- **建议**: 先运行 `ruff check --fix` 自动修复，再手动审查剩余问题。

### P2-2: 废弃的 datetime.utcnow() 使用
- **来源**: BUG-002
- **问题**: `onenote.py` 中使用已废弃的 `datetime.utcnow()`（Python 3.12+废弃）。
- **影响**: 未来Python版本将移除此API；返回的是naive datetime，可能导致时区相关bug。
- **修复**: 替换为 `datetime.now(datetime.UTC)`。

### P2-3: 类型错误 — 缺少Optional导入
- **来源**: BUG-003
- **问题**: `progress.py` 中使用 `Optional` 类型注解但未导入。
- **影响**: 运行时类型检查或使用mypy时将报错。

### P2-4: 类型不匹配 — summary_evaluator.py
- **来源**: BUG-004
- **问题**: `summary_evaluator.py` 中存在类型不匹配（传入参数类型与声明不符）。
- **影响**: 可能导致运行时TypeError或逻辑错误。

### P2-5: 类型错误 — secure_storage.py 和 structured_logging.py
- **来源**: BUG-005, BUG-006
- **问题**: `secure_storage.py` 和 `structured_logging.py` 中存在类型使用错误。
- **影响**: 运行时可能产生意外行为。

### P2-6: 内存缓存无上限
- **来源**: 架构翻新计划 P2-6
- **问题**: 内存缓存未设置最大容量限制。
- **影响**: 长时间运行后可能消耗过多内存，导致OOM。
- **建议**: 使用LRU缓存策略，设置 `maxsize` 参数。

### P2-7: 无类型检查强制执行
- **问题**: 项目未配置mypy/pyright进行静态类型检查，多个类型错误未被捕获。
- **影响**: 类型相关bug只能在运行时发现，增加调试成本。

### P2-8: 无CI/CD流水线
- **问题**: 项目缺少自动化CI/CD流水线（lint → typecheck → test → build）。
- **影响**: 代码质量依赖人工检查，易引入回归bug。

### P2-9: API密钥可能以明文存储
- **问题**: LLM服务（OpenAI/Anthropic/Zhipu/Ollama）的API密钥存储方式需审查，可能以明文形式存在于配置文件中。
- **影响**: 配置文件泄露即暴露所有API密钥。
- **建议**: 使用 `keyring` 库进行操作系统级安全存储。

### P2-10: 无依赖漏洞扫描
- **问题**: 未使用 `pip-audit` 或 `safety` 等工具扫描依赖库的已知漏洞。
- **影响**: 可能使用存在已知CVE的第三方库。

## 与三国杀OL的参考对比

作为一款复杂桌面应用（类似三国杀OL客户端），本项目可参考以下经验：

1. **GUI组件化**: 三国杀OL将对局界面拆分为玩家面板、手牌区、技能按钮、聊天框等独立组件 → 本项目app.py需类似拆分
2. **事件驱动架构**: 三国杀OL使用事件总线处理游戏事件（出牌、技能触发、伤害结算）→ 本项目已有领域事件机制，应扩展到GUI层
3. **插件系统**: 三国杀OL支持武将扩展包 → 本项目的LLM适配器已是插件模式，但需更完善的插件生命周期管理
4. **安全通信**: 三国杀OL对服务器通信进行加密和完整性验证 → MCP通信需类似的消息签名机制
5. **离线容错**: 三国杀OL支持断线重连 → 本项目在网络不稳定时的文章抓取/摘要生成需要更好的重试和回退策略
